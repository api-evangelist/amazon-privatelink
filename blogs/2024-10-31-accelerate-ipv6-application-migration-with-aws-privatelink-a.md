<p>In this post, we review how you can leverage AWS PrivateLink support for User Datagram Protocol (UDP) services, and accelerate Internet Protocol version 6 (IPv6) migrations with UDP support for dual stack Network Load Balancers (NLBs). We outline the steps to configure the two new features, their use cases, and best practices for integrating both in your existing environments.</p> 
<p>With the addition of UDP support, AWS PrivateLink allows you to privately connect your Amazon Virtual Private Cloud (VPC) to UDP-based services exposed through PrivateLink endpoints. These services can be deployed in your AWS environment across multiple VPCs, or shared with your AWS accounts by partners and Software as a Service (SaaS) providers. You can now create VPC Endpoints to access UDP-based VPC Endpoint Services, just like you access Transmission Control Protocol (TCP)-based services.</p> 
<p>To help you accelerate your IPv6 adoption, AWS now supports UDP listeners for dual stack NLBs. This allows you to automatically distribute incoming traffic to UDP-based applications, such as real-time gaming, voice over IP (VoIP), and media streaming, to ensure high availability and scalability. Dual stack NLBs allow both IPv4 and IPv6 clients to access your application endpoints, accelerating your IPv6 adoption journey on AWS. With UDP support for dual stack NLBs, you can benefit from the same feature available with TCP listeners, such as health checks, automatic scaling, and transparent upgrades, without the need to maintain custom load balancing solutions.</p> 
<h1>Prerequisites</h1> 
<p>In this post, we assume you are familiar with fundamental networking constructs on AWS, such as <a href="https://aws.amazon.com/vpc/">Amazon VPC</a>, <a href="https://aws.amazon.com/elasticloadbalancing/">Elastic Load Balancing</a>, <a href="https://aws.amazon.com/elasticloadbalancing/network-load-balancer/">NLBs</a>, and <a href="https://aws.amazon.com/privatelink/">AWS PrivateLink</a>, as well as <a href="https://aws.amazon.com/pm/cloudwatch/">Amazon CloudWatch</a>. We don’t focus on defining these services and concepts, but we do use them to show use cases and configuration steps for the two new features. For more IPv6 adoption resources on AWS, refer to the AWS re:Post article <a href="https://repost.aws/articles/AREREdDTYdSh2-NMBqQJS4wQ/get-started-with-ipv6-on-aws-resources-content">Get started with IPv6 on AWS – Resources &amp; Content</a>.</p> 
<h1>UDP support for dual stack NLBs and AWS PrivateLink</h1> 
<p>With UDP support for dual stack NLBs, clients can use both IPv4 and IPv6 to connect to your UDP service. To use this feature, you must create an IPv6-type target group for your dual stack NLB UDP listener. The NLB sends UDP traffic to your targets using IPv6.</p> 
<p>Clients can directly access dual stack NLBs with UDP listeners, by resolving the associated Fully Qualified Domain Name (FQDN). You can also configure AWS PrivateLink services (VPC Endpoint Services) using dual stack NLBs with UDP listeners. Consumers can create AWS PrivateLink endpoints (VPC Endpoints) in their VPCs, to access the UDP-based services, using IPv4 or IPv6.</p>

The service provider service target group must use IPv6. Consumers have the option to deploy and send traffic to IPv4, IPv6, and dual stack VPC Endpoints. Clients can then connect to the VPC Endpoint to use the UDP-based service.</p>

<h1>Source IP address preservation for dual stack NLBs</h1> 
<h3><strong>TCP and TLS listeners</strong></h3> 
<p>When IPv4 clients directly connect to a dual stack NLB with IPv6 targets, the NLB cannot preserve the original IPv4 address of the client. This is because IPv6 is not backwards compatible with IPv4. IPv6 targets cannot process the IPv4 address of the clients, so the NLB translates the client IPv4 source address to its own IPv6 address, to match the IP version of the targets.</p> 
<p>Similarly, when IPv4 and IPv6 clients connect to a VPC Endpoint, the NLB does not preserve the original IP address of the client, regardless of the protocol version. Since the NLB performs IP NAT, you can use AWS PrivateLink regardless of any overlapping IP addresses between consumer and service provider VPCs. As a result of the NAT, the target application is not aware of the client’s original IP address. To let the target application know the original client IP address, you can enable Proxy Protocol version 2 (PPv2) on the target group. This allows the NLB to add the client IP address as a Type-Length-Value (TLV) in the packet header, and can be enabled for both IPv4 and IPv6-type target groups.</p> 
<h3><strong>UDP and TCP_UDP listeners</strong></h3> 
<p>UDP is a connectionless protocol, therefore source NAT requires a different implementation to overcome this challenge with UDP support for dual stack NLBs. There are two key differences compared to TCP:</p> 
<ol> 
 <li>In UDP, unlike TCP, there is no concept of a connection, so ports cannot be used to track sessions through the NLB. Thus, NLB uses IPv6 addresses to uniquely identify each client of a UDP listener.</li> 
 <li>For TCP traffic, NLB adds the PPv2 header added only to the first packet, when the connection is established. In UDP, there is no session being established, and each packet is viewed as independent. As a result, when PPv2 is enabled on a UDP target group, the NLB adds the PPv2 header to every packet.</li> 
</ol> 
<p>Let’s review the configuration steps for UDP listeners on dual stack NLBs, and UDP-based VPC Endpoint Services.</p> 
<h1>Configuring UDP listeners for dual stack Network Load Balancers</h1> 
<p>Before you begin, ensure you meet the following prerequisites:</p> 
<ul> 
 <li><strong>Verify your VPC IP Configuration:</strong> Confirm your VPC has an associated IPv6 prefix. For more details on configuring IPv6 on your VPC, refer to <a href="https://docs.aws.amazon.com/vpc/latest/userguide/vpc-migrate-ipv6-add.html">Add IPv6 support for your VPC</a> in the <a href="https://docs.aws.amazon.com/vpc/latest/userguide/what-is-amazon-vpc.html">Amazon VPC User Guide</a>.</li> 
 <li><strong>Verify your subnet IP Configuration:</strong> The NLB subnets must be dual stack. The target group subnets can be dual stack or IPv6-only.</li> 
 <li><strong>Verify your Amazon Elastic Compute Cloud (Amazon EC2) instance configuration:</strong> Each instance must have a primary IPv6 address configured. For details on configuring primary IPv6 addresses, refer to <a href="https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/managing-network-interface-ip-addresses.html">Manage the IP addresses for your network interface</a> in the <a href="https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/concepts.html">Amazon EC2 User Guide</a>.</li> 
 <li><strong>Create security group ingress rules for your NLB and targets:</strong> Your security group ingress rules must allow UDP on the port(s) your NLB UDP listener and your targets are configured to receive traffic on.</li> 
 <li><strong>Ensure your application allows IPv6 traffic:</strong> Confirm your application can process IPv6 traffic, and uses a socket library supporting IPv6.</li> 
</ul>

<h1><strong>Monitor your UDP-based services with Amazon CloudWatch</strong></h1> 
<p>After you complete this configuration, consumers can send traffic to their UDP-based services. Service owners can monitor the health and performance with Amazon CloudWatch metrics such as:</p> 
<ul> 
 <li><strong>NewConnectionCount:</strong> Measures the number of new UDP flows through the NLB.</li> 
 <li><strong>PacketsOutToTarget:</strong> Measures the number of UDP packets sent from the NLB to the registered targets.</li> 
 <li><strong>HealthyHostCount:</strong> Measures the number of healthy targets in the target group.</li> 
</ul> 
<p>These CloudWatch metrics can help you ensure your UDP-based services are serving traffic as expected, and identify any potential issues. For more information, refer to <a href="https://docs.aws.amazon.com/vpc/latest/privatelink/privatelink-cloudwatch-metrics.html">CloudWatch metrics for AWS PrivateLink</a> in the AWS PrivateLink User Guide.</p> 
<h1><strong>Considerations</strong></h1> 
<p>Dual stack UDP NLBs retain existing functionality. They support one VPC Endpoint Service with many NLBs. This includes the ability to send traffic to on-premises targets. VPC Endpoint Services serving dual stack NLBs with UDP listeners support endpoint policies, private DNS names, and security groups. AWS PrivateLink throughput, service quotas, and source IP preservation remain unchanged with the release of UDP support.</p> 
<h1><strong>Conclusion</strong></h1> 
<p>In this post, we reviewed how you can leverage AWS PrivateLink support for UDP services, and how you can accelerate IPv6 migrations with UDP support for dual stack NLBs. We outlined the steps to configure the two new features, their use cases, and best practices for integrating both in your existing environments. These features help you accelerate your IPv6 adoption and migration of UDP-based services to AWS, and automatically distribute incoming traffic to UDP-based applications, such as real-time gaming, voice over IP (VoIP), and media streaming. With UDP support for dual stack NLBs, you can benefit from the same feature available with TCP listeners, such as health checks, automatic scaling, and transparent upgrades, without the need to maintain custom load balancing solutions.</p>

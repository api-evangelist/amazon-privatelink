<p>In our previous post&nbsp;<a href="https://aws.amazon.com/blogs/networking-and-content-delivery/external-connectivity-to-amazon-vpc-lattice/">External Connectivity to Amazon VPC Lattice</a>, we discussed how your applications could consume <a href="https://docs.aws.amazon.com/vpc-lattice/latest/ug/services.html">VPC Lattice services</a> from clients external to Amazon Web Services (AWS), as well as cross-<a href="https://aws.amazon.com/about-aws/global-infrastructure/regions_az/">Region</a> patterns. At re:Invent 2024, AWS made three significant announcements in the application networking space: releasing the&nbsp;<a href="https://aws.amazon.com/about-aws/whats-new/2024/12/access-vpc-resources-aws-privatelink/">VPC resources</a>&nbsp;capability for&nbsp;<a href="https://aws.amazon.com/privatelink/">AWS PrivateLink</a>, support for accessing these resources through <a href="https://aws.amazon.com/about-aws/whats-new/2024/12/vpc-lattice-tcp-vpc-resources/">Amazon VPC Lattice</a>, and the capability to create&nbsp;<a href="https://docs.aws.amazon.com/vpc-lattice/latest/ug/vpc-interface-endpoints.html">endpoints for VPC Lattice service networks</a>.</p> 
<p>These releases create opportunities for extra functionality and streamlining of your multi-modal service-to-service communications. Although the new connectivity patterns are explained in detail in the&nbsp;<a href="https://aws.amazon.com/blogs/networking-and-content-delivery/amazon-vpc-vattice-modernize-and-simplify-your-enterprise-network-architectures/">Amazon VPC Lattice: modernize and simplify your enterprise network architectures</a> post, we wanted to spend some time discussing the following points in more detail:</p> 
<ol> 
 <li>What impact does the implementation of these new components have on service endpoint discovery?</li> 
 <li>How should I configure DNS resolution to consume my services/resources?</li> 
</ol> 
<p>In this post, we focus on how to obtain the AWS-generated domain name configuration for VPC Lattice services and VPC resources, and how to configure DNS resolution properly for your consumers.</p> 
<p>Ready?</p> 
<h2>Brief reminder: how does a VPC Lattice service network VPC association work?</h2> 
<p>We start with a refresher for VPC Lattice, using the architecture in Figure 1. This shows a consumer VPC, directly associated with a service network (standard architecture prior to re:Invent 2024).</p> 
<p>When you create a VPC Lattice service you are provided with an AWS-generated DNS name that represents it. This name is globally unique and externally resolvable to a series of IP addresses in the <strong>169.254.171.x/24</strong> range and the <strong>fd00:ec2:80::/64</strong> range. IPv4 Link-local and IPv6 unique local addresses are not globally routable and are intended for devices that are either connected to the same physical (or logical) link or within a more limited area/site. Within AWS, when an application in a&nbsp;consumer&nbsp;VPC resolves a VPC Lattice service to these addresses, packets put on the wire to that address and are routed to a hidden&nbsp;ingress&nbsp;endpoint for VPC Lattice.</p>

 <h2>Accessing VPC resources through Associations and Endpoints</h2> 
 <p>The recap is complete, now we can dive into what’s new here.</p> 
 <p>VPC Resources offers a new data-plane within AWS, where endpoints such as <a href="https://aws.amazon.com/products/databases/">AWS Databases</a>, <a href="https://aws.amazon.com/ec2/">Amazon Elastic Compute Cloud (Amazon EC2)</a> instances, Domain Name targets, and IP addresses can be reached. Both PrivateLink and VPC Lattice now support direct TCP connections to endpoints. This eliminates the requirement for <a href="https://docs.aws.amazon.com/vpc/latest/privatelink/privatelink-share-your-services.html">AWS Network Load Balancers</a> (NLBs) with PrivateLink. For brevity we won’t cover the details on how VPC resources work here, instead we recommend that you check out the details in <a href="https://aws.amazon.com/blogs/aws/securely-share-aws-resources-across-vpc-and-account-boundaries-with-privatelink-vpc-lattice-eventbridge-and-step-functions/">Jeff Barr’s News post</a> to get an overview of the new features.</p> 
 <p>However, we can summarize some important concepts before discussing the DNS resolution:</p> 
 <ol> 
  <li>Resource definition: VPC resources are defined by creating a <a href="https://docs.aws.amazon.com/vpc/latest/privatelink/resource-configuration.html">Resource Configuration</a> that can be either: 
   <ol> 
    <li>Single (single resource with a target type DNS or IP)</li> 
    <li>Group (collection of child resource configurations)</li> 
    <li>Child (object within a group, sharing the same configuration)</li> 
    <li>ARN (to access supported AWS resources).</li> 
   </ol> </li> 
  <li>Resource Association: VPC resources can be exposed using three methods: 
   <ol> 
    <li>Association with an Interface endpoint of the type “<a href="https://docs.aws.amazon.com/vpc/latest/privatelink/use-resource-endpoint.html">resource</a>”</li> 
    <li>Association with an Interface endpoint of the type “<a href="https://docs.aws.amazon.com/vpc/latest/privatelink/access-with-service-network-endpoint.html#create-service-network-endpoint">service network</a>”</li> 
    <li>Association with a VPC Lattice <a href="https://docs.aws.amazon.com/vpc-lattice/latest/ug/service-network-associations.html#service-network-resource-config-associations">service network</a></li> 
   </ol> </li> 
 </ol>

 <h3>Accessing VPC resources through a VPC Lattice service network VPC association</h3> 
 <p>In this configuration, a consumer VPC is associated with a service network that also has VPC resources associated to it. How does this VPC resource publication differ from VPC Lattice services?</p> 
 <p>As mentioned previously, a globally unique and externally resolvable domain name is created per resource association to the service network as well as a Hosted Zone ID. Depending on the Resource Configuration type, you find some differences:</p> 
 <ul> 
  <li>Child resources (regardless of being in a Group or not) have their own resolution configuration</li> 
  <li>ARN resources create a managed Child resource configuration, which is the one targeting the resource itself</li> 
 </ul> 
 <p>One key difference between VPC Lattice services and VPC resources DNS resolution is in the IP address space used. The IPv4 addresses used for resource connectivity come from the public <a href="https://docs.aws.amazon.com/vpc/latest/privatelink/resource-configuration.html#service-network-types">129.224.0.x/17</a> range. For IPv6, addresses come from the fd00:ec2:80::/64 range – used for both VPC Lattice services and VPC resources.</p>

 <h3>Accessing VPC Lattice services and VPC resources through VPC Lattice service network endpoints</h3> 
 <p>As mentioned previously, VPC Lattice services and VPC resources can now be reached by using VPC Lattice service network endpoints. Service network endpoints now work beyond their associated VPCs, enabling connections from on-premises locations and cross-Region applications. These endpoints support IPv4, IPv6, and dual-stack addressing. For IPv6 addresses the <a href="https://docs.aws.amazon.com/AWSEC2/latest/APIReference/API_NetworkInterface.html">denyAllIgwTraffic</a>&nbsp;flag is enabled, which means that the endpoint can only be used for private communication.</p> 
 <p>A typical hybrid model using this feature is shown in Figure 9.</p>

 <p>If you are familiar with <a href="https://docs.aws.amazon.com/vpc/latest/privatelink/create-interface-endpoint.html">interface VPC endpoints</a>, then we should discuss some differences:</p> 
 <ul> 
  <li>With Interface Endpoints of the type ‘service network’, you don’t need to create an endpoint per service/resource to connect to. All of the services/resources associated can be reached from a single endpoint.</li> 
  <li>Each endpoint can consume multiple /28 IPv4 and /80 IPv6 range per subnet/AZ. From this range, several VPC Lattice services may consume and share a single secondary IP address, while each VPC resource consumes independent secondary IP addresses.</li> 
 </ul> 
 <p>This means that the Primary interface IP address of the interface cannot be used for routing traffic for services or resources. They must be resolved to their respective secondary IP addresses.</p> 
 <p>A new globally unique and externally resolvable domain name is created for each service/resource associated to the service network associated to the endpoint. These domain names provide the IP addresses each service/resource is consuming from the secondary range. They are viewable in the console, when looking at the <strong>Associations</strong> tab of the endpoint.</p>

 <h2>Service connectivity from external applications</h2> 
 <p>So far we have covered the five connectivity patterns documented within Table 1. However, you may be missing one: how to consume services/resources from consumers outside of AWS without a private communication. As new services/resources are added to your environment, IP addresses that are used to connect to them can change. For this reason, our view is that it remains clearer to harness an ingress proxy layer that can perform dynamic lookups and resolution for your services and resources on your behalf. Without this dynamic layer front-ending your traffic, further steps would be needed to discover endpoint secondary IP address information and dynamically update any static configurations that you have deployed.</p> 
 <p>The solution you choose can be a wholly owned one (as discussed in our previous post) or might involve a partner offering. Ingress proxying for VPC Lattice services is also possible using <a href="https://aws.amazon.com/api-gateway/">Amazon API Gateway</a>, as discussed in this <a href="https://aws.amazon.com/blogs/containers/securing-api-endpoints-using-amazon-api-gateway-and-amazon-vpc-lattice/">Amazon Containers post</a>.</p> 
 <p>Check the <a href="https://aws.amazon.com/solutions/guidance/external-connectivity-to-amazon-vpc-lattice/">Guidance for External Connectivity to Amazon VPC Lattice</a> for a fully automated solution using an <a href="https://docs.aws.amazon.com/elasticloadbalancing/latest/network/introduction.html">NLB</a> targeting a fleet of lightweight serverless proxies on <a href="https://docs.aws.amazon.com/AmazonECS/latest/developerguide/AWS_Fargate.html">AWS Fargate</a>.</p>

 <p>With the release of VPC Lattice service network endpoints, this solution can be further optimized. Previously there was a 1:1 mapping consideration between the ingress solution VPC and the service network with which it was associated. Opting to use an endpoint instead of a VPC association allows this ingress proxy solution to be used to reach any service network within the Region!</p> 
 <p>PrivateLink resource endpoints allow you to remove the need of the proxy solution and target endpoints directly from an ELB. At scale, we still recommend the use of a VPC Lattice service network to have a single place to control the connectivity and security of your services/resources and reduce endpoint sprawl.</p>

 <h2>Further considerations</h2> 
 <ul> 
  <li>Consider the use of VPC Lattice service network associations as the principal access method for both VPC Lattice service and VPC resource access. This streamlines the connectivity requirements and creates a strong mapping between a VPC and its intended use.</li> 
  <li>Consider using VPC Lattice service network endpoints for multi-dimensional use cases such as split function VPCs, hybrid and external access through proxy layers.</li> 
  <li>Consider the use of PrivateLink resource endpoints when you are predominantly an PrivateLink user with no immediate plans for scaling access to services.</li> 
  <li>In multi-Account environments, you can share the following resources through <a href="https://docs.aws.amazon.com/vpc-lattice/latest/ug/sharing.html">AWS Resource Access Manager</a> (AWS RAM): VPC Lattice service networks, VPC Lattice services, and VPC Resource Configurations.</li> 
  <li>When working at scale, we recommend the following related to the DNS resolution configuration: 
   <ul> 
    <li>The automation of the DNS configuration to avoid bottlenecks anytime a new VPC Lattice service or VPC resource is created/deleted. The <a href="https://aws.amazon.com/solutions/guidance/amazon-vpc-lattice-automated-dns-configuration-on-aws/">Guidance for Amazon VPC Lattice Automated DNS Configuration on AWS</a> provides a solution you can use to automate these DNS configurations at scale.</li> 
    <li>The use of <a href="https://docs.aws.amazon.com/Route53/latest/DeveloperGuide/profiles.html">Amazon Route 53 Profiles</a> to streamline the association of Private Hosted Zones to consumer VPCs.</li> 
   </ul> </li> 
  <li>You can apply granular security with VPC Lattice services using <a href="https://docs.aws.amazon.com/vpc-lattice/latest/ug/sigv4-authenticated-requests.html">sigV4 signing</a>. For applications that reside outside of AWS, consider using <a href="https://docs.aws.amazon.com/rolesanywhere/latest/userguide/introduction.html">IAM Roles Anywhere</a> to get the necessary credentials.</li> 
 </ul>

We provided some guidance on how to think about the new AWS capabilities and where and how to deploy and access them.</p>
